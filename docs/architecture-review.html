SBCO BACnet B-BC Simulator / Gateway

実設備なしで、SBCO 標準ポイントリスト 1 つから標準準拠の仮想 B-BC を BACnet/IP（北向き）で公開し、 南向き（MQTT/ZeroMQ/WoT/gRPC）データ源を BACnet 化して上位（Hono → Building OS）へ供給する。

Python 3.12 · uv BACnet lib bacpypes3 concurrency single-loop asyncio ~2,800 LOC src 16 test files · 120 passing ruff + mypy clean MVP-1/2 done · MVP-3 partial

Executive Summary

本リポジトリは設計ドキュメント群から出発し、6 エピック（EP-001〜006）を TDD ＋ エピック単位レビューで実装し、全て main にマージ済み。MVP-1/MVP-2 を完成、MVP-3 は 成果物出力を中心に partial で着地している。

何を作ったか

SBCO CSV → YAML 中間モデル → bacpypes3 オブジェクト → BACnet/IP 公開。Simulator / Gateway / Combined の 3 動作モード、南向きバインディング、 値生成・フォールト注入、COV、REST 制御面、意味タグ、標準成果物エクスポートまで。

なぜ重要か

実 B-BC は開発初期に存在せず・調達高・CI に乗らない・異常系再現が困難。 本製品はその代替＋プロトコル変換ハブとして、接続ゲートウェイ／Building OS の結合試験を CI まで含めて成立させる。

現在地

完了: 北向き全サービス・南向き抽象＋MQTT/ZeroMQ・ シミュレーション・上位連携(BBMD/FDR)・意味タグ・エクスポート。 将来: BACnet/SC・WoT南向き・将来オブジェクト（issue #28/#30/#31）。

健全性
ループバック統合テストで BACnet/IP プロトコル往復（Who-Is/Read/RPM/Write/COV）を実機の bacpypes3 スタック上で検証。CI は外部ブローカ不要の in-repo フェイクで無人グリーン、実ブローカは integration ジョブに分離。

System Context

北向きは常に BACnet/IP。上位系（Building OS）は接続ゲートウェイ（Eclipse Hono 等）経由で 取り込む。製品は上位へ直結しない（ADR-005）。

Northbound = BACnet/IP（出力・常時） Southbound = binding（入力・Gateway時） 本製品スコープ

Architecture & Pipeline

全モードが YAML 由来の単一オブジェクトモデルを共有（ADR-004）。 並行性は single-loop asyncio（ADR-010）で、Core Object Model は event loop に閉じ込める。

Northbound Adapter は値の出所を意識せず Core のみ参照。値更新経路は simulate→Core または southbound telemetry→Core。command 経路は 北 WriteProperty→Core→（writable かつ binding あれば）southbound 送出。

Data & Control Flow

Telemetry（南 → 北）gateway/combined

  1. Transport がチャネル受信（address: local_id 優先で導出）
  2. mapping.telemetry_to_present_value で型別正規化（scale/offset/enum）
  3. Core の presentValue を更新（event loop 上）
  4. 北向き ReadProperty / COV Notify で上位が取得

Command（北 → 南）writable のみ

  1. 北向き WriteProperty 受信（BBCApplication）
  2. writable 検査 — Input は writeAccessDenied で拒否
  3. 成功時 command フック発火（command-bound オブジェクトのみ）
  4. mapping.present_value_to_command → Transport へ publish
論理一致（AC-13 / PR-NF-017）
同一オブジェクトの南向き入力値と北向き presentValue が論理一致することを、実オブジェクトモデル上の ループバックテストで検証済み。

Key Design Decisions (ADRs)

設計判断の正は docs/adr/。特に方向性と並行性の 2 点は全体を貫く前提。

ADR決定影響
ADR-001SBCO ポイントリストを唯一の入力ソースに入力境界の単純化
ADR-003gateway_id ≠ bbc_id（混同禁止）識別子の安全性
ADR-004YAML を中間モデルとし全モードで共有モード間整合
ADR-005北向き=BACnet/IP、南向き=binding（v1.2 訂正）方向の正
ADR-007object type を BACnet 列優先＋推定（警告付き）マッピング規則
ADR-008Raspberry Pi ネイティブ実行を一級・Docker 任意配布
ADR-009BACnet ライブラリを bacpypes3 に一本化技術選定
ADR-010single-loop asyncio・Core を loop 閉じ込め並行性の正
ADR-011device-mapping mode（aggregated/multi/auto）。runtime mode と直交用語/採番
ADR-012device_type=Brick クラス採用、タグは Brick 由来。SBCO tags 列は検索タグで別物意味付け
ADR-013南向き Transport 抽象（InMemory/MQTT/ZeroMQ）プロトコル拡張性
2 つの "mode" 軸を混同しない
runtime mode（値の出所: simulator/gateway/combined）と device-mapping mode （SBCO device→BACnet device 写像: aggregated/multi-device/auto-partition）は直交。 MVP-1 は aggregated（1 Virtual B-BC に集約）。

Non-negotiable Invariants（コードで担保）

北向き = BACnet/IP のみ
南向きは決して出力にしない。network.type=bacnet-ip 固定、MQTT 等は入力源。検証: 全suite・方向定義 ADR-005
gateway_id ≠ bbc_id
bbc_id/device_id は CLI 引数由来。gateway_id は metadata のみ。検証: test_generator.py
1 ランタイム = 1 Virtual B-BC
aggregated で全点を 1 Device に集約。検証: test_generator.py
SBCO のみ入力 / YAML が中間
入力は SBCO 1 種、全モードが simulator.yaml を共有。検証: pipeline 全体

Risks & Gaps

bacpypes3 0.0.106: WPM エラー応答が IP で伝送できないMEDIUM
WritePropertyMultiple の成功は往復するが、WritePropertyMultipleError（write-access-denied 等）の 応答 PDU を IP データリンクが送出できずクライアントがタイムアウトするライブラリ側の不具合。
対応: WPM の writable 強制はハンドラレベルで検証（正しいエラー型を raise）。単発 WriteProperty の エラーは正常に伝送。記録: docs/memory/pitfalls.md
REST 制御面に認証なしLOW
127.0.0.1 バインドの運用/テスト用サイドチャネル。present-value 書込・フォールト注入が可能。 リモート公開する場合は認証が必要。北向きは BACnet/IP のみで影響は局所。
Brick→Haystack seed テーブルは暫定LOW
決定的だが網羅は限定的。確定版マッピング CSV 待ち（ADR-012 の open ❓）。name:value タグ （unit:"°C" 等）は未対応（marker のみ）。
外部システム連携は手動 ACMEDIUM
Hono / Ditto / Building OS の実到達（AC-7/8）、実 BBMD を挟む異サブネット探索（AC-10）、ARM 実機起動、 YABE GUI 疎通は自動化対象外。CI/loopback では「BACnet クライアントが全オブジェクト探索・読取」で 取り込み可能性を代理確認。記録: README "Manual acceptance"
MVP-3 未実装（将来）— issue #28/#30/#31 を OPEN 維持SCOPE
BACnet/SC・BTL 適合支援（#31）、WoT 南向きバインディング（#30）、将来オブジェクト Schedule/TrendLog/Calendar/NotificationClass/Accumulator（#28）、QUDT/full-REC エクスポートは未実装。 成果物エクスポート（EDE/IEIEJ/PICS/JSON-LD/WoT-TD）は実装済み。
テスト環境の留意点LOW
bacpypes3 オブジェクト構築には実行中の event loop が必要（sync テストは conftest の autouse fixture で供給）。 実ブローカ統合は @pytest.mark.integration で既定スキップ。
